[CLICK] Pass 0: regex. Remember those free facts from the LitRPG prose? We grab them here. Zero cost, 100% confidence. Facts are in the graph. Now we need to check them. We let technical users — typically platform admins — write Cypher queries directly and save them as reusable rules. But authors aren't writing Cypher. So we let them describe rules in plain English — "flag me if a character's eye color changes" — and Neo4j's Text2Cypher translates that into Cypher. [CLICK] Either way, the rules get saved and run against the graph. Let me show you what some of these rules look like.

Has This Character's Eye Color Changed?

-- Cypher: detect_fact_conflicts()

MATCH (f1:Fact), (f2:Fact)
WHERE f1.entity_name = f2.entity_name
  AND f1.fact_key    = f2.fact_key
  AND toLower(f1.fact_value) <> toLower(f2.fact_value)
  AND f1.is_stale = false
  AND f2.is_stale = false
  AND id(f1) < id(f2)
RETURN f1.entity_name as entity,
       f1.fact_key as fact_type,
       collect(DISTINCT f1.fact_value)
         + collect(DISTINCT f2.fact_value) as values
entityfact_typevalues
elena blackwoodeye_color["blue", "green", "brown"]
elena blackwoodstaff_material["oak", "maple", "pine"]

A continuity error detector in 12 lines of Cypher.

Here's what a rule looks like. Twelve lines of Cypher — find any two non-stale facts about the same entity and attribute with different values. [CLICK] Run it — Elena's eyes went from blue to green to brown. Staff went from oak to maple to pine. A continuity error detector you can write in minutes. What about temporal rules?

Stats Changed Without a Reason?

MATCH (c:Character)-[s1:HAS_STAT {name: 'strength'}]->(scene1:Scene),
      (c)-[s2:HAS_STAT {name: 'strength'}]->(scene2:Scene)
WHERE scene2.sequence = scene1.sequence + 1
  AND s1.value <> s2.value
  AND NOT EXISTS {
    MATCH (c)-[:PARTICIPATES_IN]->(e:Event)
          -[:MODIFIES_STAT]->(c)
    WHERE e.scene = scene2.id AND e.stat = 'strength'
  }
RETURN c.name, s1.value, s2.value,
       scene1.title, scene2.title

Strength went from 12 to 15 between scenes — but nothing happened to cause it.

This one's my favorite for the Neo4j crowd. The query checks consecutive scenes for stat changes that have no triggering event in the graph. [CLICK] Strength jumped from 12 to 15 — but nothing in the event chain explains why. Here's another example from our test data.

"The graph is a living document, not a one-time extraction."

So the graph has to keep up. We mark facts from the edited chapter as stale, re-extract, then clean up. But the re-extraction step is a trade-off. [CLICK] Incremental: just re-extract the chapter that changed. Fast, cheap, do it on every save. But you might miss cross-chapter impacts — an edit in chapter 5 could contradict something in chapter 2. Full: re-extract the entire book. Expensive — remember, 3-4 LLM calls per chunk — but guaranteed consistency. You do this at milestones. Draft complete, structural rewrite, before sending to an editor. When the author saves, a validation agent kicks off. It does two things. First: the static rules we just saw — the saved Cypher queries — run automatically against the graph. Eye color drift, dead characters, missing limbs. The known problems. [CLICK] Second — and this is where it gets interesting — the LLM reads what the author just wrote, looks at the graph schema, and generates its own Cypher queries via Text2Cypher. It checks things you didn't think to write rules for. The unknown unknowns. [CLICK] Both layers hit Neo4j. Contradictions come back as inline comments. The graph turns 500K words of novel into about 2KB of targeted context for the LLM. And these patterns aren't limited to fiction.

What's Hard (Honestly)

Entity resolution

"the healer" = "Lyra" = "she" = "her" — below 85% accuracy, the whole graph is unreliable

Extraction cost

3-4 LLM calls per chunk per chapter — not cheap at scale

False positives

The Tom twin scenario — intentional plot twists get flagged

Sync during rapid edits

Authors rewrite fast — incremental vs full re-extract is a real trade-off

"The graph doesn't replace human judgment. It amplifies it."

Now let me be honest about what's hard — because if I don't tell you, you'll find out the hard way. Entity resolution is the one that'll hurt you first. "The healer," "Lyra," "she," "her" — if you can't resolve those below 85% accuracy, the graph is full of orphan facts and everything downstream is wrong. Extraction is expensive. False positives are annoying — the system will flag your clever plot twists. And keeping the graph current when an author is rewriting furiously? Still an unsolved UX problem. [CLICK] But here's the thing. The graph doesn't replace human judgment. It amplifies it. And that's enough.
